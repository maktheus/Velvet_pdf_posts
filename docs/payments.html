arquitetura

O sistema de pagamentos usa o Strategy Pattern: cada gateway é uma implementação de PaymentGateway. O OrderService nunca conhece o gateway — apenas chama a factory. Trocar ou adicionar um gateway é zero impacto no restante do sistema.

PaymentGateway.java — interface principal

// Interface (nunca muda quando adicionar novo gateway)
public interface PaymentGateway {
    PaymentResponse  createPayment(PaymentRequest request);
    PaymentStatus    getStatus(String externalId);
    RefundResponse   refund(String externalId, BigDecimal amount);
    void             processWebhook(String payload, String signature);
    String           provider();
}

PaymentGatewayFactory.java — Spring injeta todos os gateways automaticamente

@Service
public class PaymentGatewayFactory {
    private final Map<String, PaymentGateway> gateways;

    public PaymentGatewayFactory(List<PaymentGateway> all) {
        this.gateways = all.stream()
            .collect(Collectors.toMap(
                PaymentGateway::provider,
                g -> g
            ));
    }

    public PaymentGateway get(String provider) {
        return Optional.ofNullable(gateways.get(provider))
            .orElseThrow(() -> new UnsupportedGatewayException(provider));
    }
}

StripeGateway.java — adicionar Stripe amanhã: zero mudança em OrderService

mercado pago — atual

Gateway ativo em produção. Três métodos de pagamento suportados.

⚡
Pix
5% de desconto automático aplicado no PaymentRequest.
💳
Cartão de Crédito
Parcelamento em até 4x sem juros configurado via preference.
📄
Boleto
Vencimento em 3 dias úteis. URL do boleto retornada no response.

webhook events tratados

payment.created Pedido registrado, aguarda confirmação
payment.approved Pagamento aprovado — dispara fulfillment
payment.rejected Pagamento recusado — status do pedido atualizado
payment.refunded Reembolso processado pelo gateway

adicionar novo gateway

1
Criar a classe do gateway
Criar XyzGateway implements PaymentGateway no pacote infra.payment. Anotar com @Component("xyz").
2
Implementar os 4 métodos
createPayment, getStatus, refund e processWebhook. Mapear modelos do provider para os DTOs internos.
3
Adicionar variáveis de ambiente
Credenciais do novo gateway no application.yml e documentar na tabela de env vars do backend.
4
Registrar endpoint de webhook
O PaymentController roteia para o gateway correto via provider. Nenhuma mudança no controller — apenas configurar a URL no painel do provider.
5
Testes de integração
Implementar XyzGatewayIntegrationTest com mocks do provider SDK. Testar todos os cenários de webhook.

paymentrequest dto

DTO de entrada para PaymentGateway.createPayment(). Agnóstico ao provider.

Campo Tipo Obrigatório Descrição
orderId String ✅ ID interno do pedido para correlação
amount BigDecimal ✅ Valor em reais (ex: 149.90)
method PaymentMethod ✅ Enum: PIX, CREDIT_CARD, BOLETO
customer CustomerDTO ✅ Nome, e-mail, CPF do comprador
items List<ItemDTO> ✅ Itens do pedido com título, qty e unit price
cardToken String ⚪ Token do cartão (gerado pelo SDK JS do gateway)
installments Integer ⚪ Número de parcelas. Padrão: 1
metadata Map<String,String> ⚪ Dados extras para rastreamento (ex: UTM source)
